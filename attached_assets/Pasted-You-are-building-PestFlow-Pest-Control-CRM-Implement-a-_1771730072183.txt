You are building PestFlow (Pest Control CRM). Implement a clean multi-location Customer detail UX where the top of the page shows brief Customer info, and all operational tabs (Contacts/Schedule/Services/Invoices/Comms) are location-scoped based on a selected Location.

GOALS
1) Keep the Customer header brief (no “relationship header” paragraph lists).
2) Make it easy to manage Customers with multiple service locations AND flexible billing (default vs per-location).
3) Support customer-wide notes (“Shared Notes”) and location-only notes with clear UI behavior.

DATA MODEL (implement in DB + API)
- Customer (id, name, type: Residential|Commercial, status, email, phone, defaultBillingProfileId nullable)
- Location (id, customerId, name/label, address fields, isPrimary boolean, gateCode, sqft, notesSummary optional, billingProfileId nullable)
- BillingProfile (id, customerId, label, methodType: Card|ACH|Invoice, token/placeholder fields, isDefault)
- Note (id, customerId nullable, locationId nullable, scope: CUSTOMER|LOCATION, pinned boolean, createdAt, createdBy, body)
Rules:
- A Customer can have many Locations.
- Exactly 0..1 primary location per Customer (enforce in service logic).
- Customer has 0..1 default BillingProfile.
- Each Location may override billing with its own billingProfileId.
- Notes:
  - CUSTOMER notes attach to customerId and show as “Shared Notes”.
  - LOCATION notes attach to locationId and show as “Location Notes”.
  - Provide an action to “Make Shared” (convert a LOCATION note into a CUSTOMER note) and “Make Location-only” (convert CUSTOMER note into LOCATION note for current location).

CUSTOMER DETAIL PAGE UX (match the annotated screenshot concept)
Layout:
A) Top header (brief customer info)
- Customer name + status badge
- Small tiles: Email, Phone, Type
- Add small chips under name:
  - Primary: <Primary Location name> (if exists)
  - Billing: Default (if all locations use default) OR “Per-location” (if any location has billingProfileId override)

B) Customer Notes panel (collapsible)
- Card titled “Customer Notes” (these are Shared Notes)
- Show first ~2 lines of the most recent pinned note (or most recent note) with an Expand/Collapse
- Buttons: “+ Add Shared Note”
- Optional: show pinned notes first

C) Location selector in the tab row area
- The “Locations (N)” tab should open a dropdown selector listing locations:
  - show location label, city/state, badges for Primary and Billing Override
- Selecting a location sets it as the active context (activeLocationId)
- The other tabs (Contacts, Schedule, Services, Invoices, Comms) should load data for the active location ONLY
- Persist activeLocationId in the URL querystring (?locationId=) and default to primary location, else first location

D) Location details area
- Show Location Profile card for the selected location (address, gate code, sqft, etc.)
- Show Location Notes card (location-only notes)
  - “+ Add Location Note”
  - Each note row has actions:
    - “Make Shared” (convert to CUSTOMER note)
- Shared Notes should NOT clutter the location notes area by default (they live in the Customer Notes panel), but you may add a small “Show Shared Notes here” toggle for convenience.

ACTIONS / FEATURES
- Add Location button (creates a new location for this customer; allow setting as primary)
- Set Primary Location action (only one per customer)
- Billing override indicator:
  - If active location has billingProfileId override, show “Billing Override” badge in the location profile card
- Basic CRUD for notes (create/list; edit/delete optional if time)

IMPLEMENTATION NOTES
- Keep the UI lightweight and intuitive. Avoid big relationship summaries.
- Use existing project stack patterns/components.
- Ensure the tab counts (e.g., Schedule (1)) reflect active location counts.
- Include loading and empty states that look clean.

DELIVERABLES
1) DB schema/migrations for the entities above (or adapt existing schema).
2) API routes/services for:
   - getCustomerById (include locations, primary, billing flags)
   - getLocationById (include location details)
   - listLocationScopedCounts (schedule/services/invoices/comms counts for active location)
   - createNote (shared or location)
   - convertNoteScope (location <-> customer)
3) CustomerDetail page updated per the UX spec.
4) Seed/dev data to demonstrate:
   - A customer with 2 locations, one primary
   - One shared note
   - One location-only note
   - One location with billing override and one without

ACCEPTANCE CRITERIA
- Customer header stays brief (name/status + email/phone/type + small chips).
- Location selection changes the data shown in Schedule/Services/Invoices/Comms tabs.
- Shared notes live at customer level in a collapsible panel; location notes live under location.
- “Make Shared” correctly moves a note from location scope to customer scope and reflects immediately in UI.
- URL retains selected location (?locationId=...).
